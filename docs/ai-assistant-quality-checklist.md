# AI 助手质量测试清单

这份文档不是讲实现细节，而是给产品验收和回归测试用。

目标只有一个：快速判断编辑器里的 AI 助手，当前到底是“会聊天”，还是“真的会操作这个 app”。

---

## 1. 先看当前产品原则

当前 AI 助手应该遵守这条分流规则：

- 在**文档工作区**，它可以明显偏创作助手
- 在**非文档界面**，它应该优先偏工作台助手

也就是：

- 在文档工作区，可以重点测写作、改写、续写、平台改写
- 在总览、Daily、白板、看板、发布、搜索、图谱、同步、设置这些界面，更该测“它会不会直接操作界面和条目”

如果这条分流失效，常见表现就是：

- 明明在总览页，它却只会给创作建议
- 明明该切界面，它却教用户自己点按钮
- 明明搜到了文档，它却不会继续打开
- 明明该新建文件或文件夹，它却只给文字方案

---

## 2. 推荐测试方法

每条 case 都按同一套方式测，避免“感觉差不多”：

1. 先切到目标界面。
2. 新开一个 AI 会话，避免上一轮上下文污染。
3. 输入测试语句，不补额外解释。
4. 记录三件事：
   - 当前界面
   - 你的原始输入
   - 实际结果
5. 只按“有没有真的落到 app 里”判定，不按回答好不好听判定。

建议记录格式：

| 项 | 内容 |
|---|---|
| 测试日期 | 2026-06-18 |
| 当前界面 | 例如：总览 / 文档工作区 / 今日速记 |
| 输入 | 原始用户话术 |
| 结果 | 实际发生了什么 |
| 判定 | 通过 / 失败 / 部分通过 |
| 备注 | 是否偏题、是否只聊天不落地 |

## 3. P0 回归点

这几类是现在最应该优先盯住的。

### 3.1 界面分流是否正确

重点判断：

- 在文档工作区，是否优先走创作动作
- 在非文档界面，是否优先走工作台动作

失败信号：

- 非文档界面里一律把需求理解成“写文章”
- 文档工作区里该改文档时，只回文本，不真正写回

### 3.2 会不会真的切界面

重点判断：

- 用户说“打开”“切到”“带我去”时，是否真的切换到对应界面

失败信号：

- 只回复“你可以点击左侧……”
- 回复正确，但页面没变化

### 3.3 搜到条目后会不会继续打开

重点判断：

- 搜索命中后，是否能继续打开文档或文件夹，而不是停在搜索结果描述

失败信号：

- 只会列文档名
- 知道有结果，但不会继续打开第一篇 / 指定那篇

### 3.4 会不会创建实际对象

重点判断：

- 新建文档是否真的创建
- 新建文件夹是否真的创建
- Daily 条目是否真的落到 Daily

失败信号：

- 只给模板文本
- 说“已创建”，但工作区里没有实际对象

### 3.5 文档改写是否走“写回 + diff”

重点判断：

- 对当前文档的改写请求，是否走实际写回链路
- 是否通过 diff 确认，而不是只在聊天区输出一版新文案

失败信号：

- 只在消息里给出改写结果
- 没有进入待确认写入状态

## 4. 核心测试清单

下面这组 case 可以覆盖当前最关键的 AI 质量问题。

| ID | 当前界面 | 测试输入 | 预期结果 | 失败信号 |
|---|---|---|---|---|
| A01 | 总览 | 带我去发布队列 | 直接切到发布队列界面 | 只解释怎么点，不切界面 |
| A02 | 总览 | 打开今日速记 | 直接切到 Daily 界面 | 回复正确但页面没动 |
| A03 | 今日速记 | 记一个今天下午两点开会 | 在当天 Daily 新增一条 event 或 note | 只回“你可以记一下” |
| A04 | 今日速记 | 先帮我记个待办：补 AI 测试文档 | 待办池新增一条 todo | 没新增，或写到聊天里 |
| A05 | 灵感白板 | 清空白板 | 白板内容被清空 | 只回复“可以清空” |
| A06 | 灵感白板 | 在白板上加三张卡片：问题、方案、验收 | 白板新增 3 张卡片 | 只输出文字列表 |
| A07 | 创作看板 | 打开一篇待发布稿 | 打开对应稿件，而不是只描述看板 | 停在口头说明 |
| A08 | 发布队列 | 带我去知识库搜索 | 切到搜索界面 | 不切界面 |
| A09 | 知识库搜索 / 图谱 | 找一下“待发布”相关稿件并打开第一篇 | 先搜到结果，再实际打开一篇文档 | 只列结果，不打开 |
| A10 | 文件夹 | 在这里建个文件夹叫灵感池 | 当前目录下出现“灵感池”文件夹 | 只给目录建议 |
| A11 | 文档工作区 | 润色第二段，让语气更紧一点 | 进入文档写回链路，出现 diff 确认 | 只在聊天区输出润色版 |
| A12 | 文档工作区 | 继续往下写一段总结 | 当前文档进入写回链路 | 只给建议，不写回 |
| A13 | 文档工作区 | 生成公众号版本，但保留原稿 | 新建一篇平台衍生稿，原文不变 | 直接覆盖当前文档 |
| A14 | 文档工作区 | 帮我开一个资料单，主题是 AI 测试 | 创建新文档，并带资料单语义 | 只返回一个大纲 |
| A15 | 文档工作区 | 有没有相关旧文可以参考 | 返回相关旧文信息，必要时附带可继续打开的对象 | 只泛泛而谈“可以查一下” |

## 6. 通过标准

建议不要只写“感觉不错”，而是按下面标准判断：

### 通过

- 理解了当前界面的任务类型
- 调用了正确动作
- 结果真实落到了 app 里
- 没有把应操作的任务退化成纯聊天

### 部分通过

- 理解方向对了
- 但只做了一半

典型例子：

- 切到了搜索界面，但没继续打开条目
- 给出了改写内容，但没写回文档
- 创建意图对了，但对象没真正出现

### 失败

- 界面分流错了
- 动作类型错了
- 没有任何真实落地

---

## 7. 典型坏味道

看到下面这些，基本可以直接记成 AI 质量问题：

- “在任何界面都先往写作方向拐”
- “只会解释，不会操作”
- “搜到了，但打不开”
- “该新建时只给模板”
- “该写回时只发一条聊天回复”
- “该保留原稿时覆盖了当前文档”

## 9. 这份清单怎么用

最实用的方式不是每次全跑一遍，而是分层：

- 改了 `agentEngine` / `toolRegistry` / `AgentPanel`：至少跑完 A01-A15
- 只改文档写作链路：重点跑 A11-A15
- 只改工作台操作链路：重点跑 A01-A10
- 发版前：挑 5 条最关键的 smoke case 跑一遍

推荐 smoke case：

- A01 总览切发布队列
- A03 Daily 新增条目
- A06 白板新增卡片
- A09 搜索并打开文档
- A13 平台版另存不覆盖原稿
