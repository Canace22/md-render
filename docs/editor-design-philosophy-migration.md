# 存量项目编辑器设计哲学复刻设计文档

## 1. 文档目标

本文档用于指导在存量项目中复刻当前编辑器的设计哲学与交互取向，目标不是简单复制一套 UI，而是把这套编辑器背后的产品判断迁移过去。

核心结论只有一句话：

**像在一张纸上写作，而不是在一个工具里操作。**

这意味着迁移时优先复刻以下能力，而不是优先复刻视觉细节：

- 正文永远是视觉中心
- 用户进入页面后可以立刻写
- 高频动作贴近内容，低频动作收纳到设置层
- 编辑器提供理解力，但不打断写作主路径
- 内容要易迁移、可导出、输出稳定

---

## 2. 复刻范围

### 2.1 需要复刻的部分

- 纸面式写作体验
- 克制的工具暴露策略
- 工作区 + 文稿 + 设置的分层结构
- Markdown 友好但不代码框化的编辑体验
- 可扩展的模式能力分层，例如小说模式这种“增强层”
- 稳定导出能力，例如微信公众号复制链路

### 2.2 不作为第一阶段目标的部分

- 1:1 还原现有样式细节
- 一次性迁移所有高级模式
- 把所有旧页面逻辑重写为编辑器原生逻辑
- 为了编辑器改造而大规模重构整站

---

## 3. 设计哲学

### 3.1 写作优先

编辑器首先服务连续表达，而不是服务工具操作。

产品要求：

- 首屏默认进入可写状态
- 不要求用户先理解“编辑态 / 预览态”
- 标题、正文、结构块都应围绕书写流组织

设计判断：

- 如果一个功能会打断写作，它应该后退
- 如果一个功能是写作中高频发生的，它应该贴近正文

### 3.2 所见即所得，但保持 Markdown 友好

这套编辑器不是纯代码编辑器，也不是重富文本后台。

目标状态：

- 交互层更像文档
- 内容层仍然可以稳定导出为 Markdown
- 粘贴、导入、导出都优先考虑 Markdown 兼容性

设计判断：

- 用户感知到的是“写文档”
- 系统内部保留“结构化文档 <-> Markdown”双向转换能力

### 3.3 界面退后，内容上前

工具必须存在，但不能长期占据注意力。

具体原则：

- 标题栏只保留文档级动作
- 块级高频操作使用轻量快捷工具条
- 低频配置统一进入设置面
- 扩展能力采用侧挂面板，而不是压在正文上方

### 3.4 纸张感是一种认知模型

“纸张感”不是白底留白这么简单，而是让用户自然把注意力投向内容。

应当具备的感受：

- 页面像稳定的写作台
- 正文像一张连续延展的纸
- 工具像附着在纸边，而不是压在纸面中央

对应的界面要求：

- 大块留白
- 弱分隔、弱卡片感
- 温和层级，不做后台管理式信息堆叠

### 3.5 低刺激，支持长时间停留

编辑器不是瞬时消费界面，需要支持长时间阅读和书写。

视觉约束：

- 正文对比清晰但不过度锐利
- 标题有层级但不过重
- 强调色只用于状态和关键操作
- 工具按钮不应比正文更“热闹”

### 3.6 高频贴近内容，低频统一收纳

推荐分层：

- 贴近正文：复制、块级插入、实体提及、轻量预览
- 进入设置：主题、导入导出、排版模板、环境偏好

这套分层的好处是：

- 保证正文附近始终干净
- 高频动作不需要跨区寻找
- 系统复杂度增加时，主界面不会失控

### 3.7 内容不是终点，输出必须稳定

编辑器不是内容终点，而是创作入口。

因此必须保证：

- 多文件切换顺滑
- 内容可落盘、可导出、可恢复
- 面向外部平台的输出稳定可控

## 4. 产品结构建议

在存量项目中建议采用三层结构，而不是把所有能力塞进一个页面容器。

### 4.1 工作区层

职责：

- 管理文件树
- 文件切换
- 新建、重命名、删除
- 设置入口

交互要求：

- 可折叠
- 不抢正文焦点
- 文件管理能力完整，但视觉弱化

### 4.2 文稿层

职责：

- 文档标题
- 正文编辑
- 正文附近高频动作
- 模式切换入口

交互要求：

- 默认进入这一层
- 正文区域始终最大
- 工具贴边，不压内容

### 4.3 扩展能力层

职责：

- 承接特定场景的结构化辅助
- 展示建议、上下文、补充信息
- 保持“可用但不强迫处理”

交互要求：

- 以侧栏或浮层存在
- 默认不抢焦点
- 不直接改正文

---

## 5. 页面与交互映射

### 5.1 推荐页面骨架

```text
工作区侧栏 | 文稿主区
           ├─ 文档标题栏
           ├─ 轻量快捷工具条
           ├─ 纸面编辑区
           └─ 可选扩展面板
```

### 5.2 推荐的 surface 状态

建议在存量项目里抽象一个 `surface` 概念，而不是到处散落弹窗状态。

建议最少包含：

- `paper`：正文写作面
- `settings`：设置面

收益：

- 文稿与设置不互相打断
- 后续增加其他 surface 时不会破坏主结构

### 5.3 推荐的 mode 概念

建议保留一个轻量 `mode`，用于承接不同写作增强模式。

建议最少包含：

- `default`
- `novel` 或其他业务增强模式

原则：

- mode 是增强，不是新编辑器
- 开启 mode 后，正文编辑路径不改变

---

## 6. 信息架构与状态模型

建议在存量项目里统一沉淀为一个编辑器 store，最少维护以下状态：

```ts
interface IEditorState {
  workspace: any;
  selectedId: string;
  markdown: string;
  surface: 'paper' | 'settings';
  mode: 'default' | 'custom';
  sidebarCollapsed: boolean;
  theme: 'light' | 'dark' | 'system';
  copyStyle: string;
}
```

### 6.1 必须统一管理的状态

- 当前工作区数据
- 当前选中文档
- 当前文稿内容
- 当前界面 surface
- 当前模式 mode
- 主题和导出模板偏好

### 6.2 必须落盘的状态

- 工作区结构
- 文稿内容
- 最近选中文档
- 用户偏好配置

### 6.3 不建议分散管理的状态

- 当前文稿与文件树选中关系
- 设置面是否打开
- 主题与导出模板
- 扩展模式是否开启

这些状态如果散落在多个页面组件里，会直接破坏迁移后的可维护性。

---

## 7. 组件分层建议

### 7.1 壳层组件

负责整体布局与模式切换，不承载具体业务细节。

建议包含：

- `WorkspaceSidebar`
- `DocHeader`
- `EditorQuickToolbar`
- `SettingsPanel`
- `PaperSurface`
- `AssistantPanel`

### 7.2 编辑器适配层

负责封装底层编辑器能力，不让业务直接耦合第三方编辑器实例。

职责：

- 初始化编辑器
- Markdown 导入导出
- 粘贴规则
- 块级操作调用
- 事件桥接

### 7.3 业务增强层

负责小说模式、知识模式、结构化提示这类业务能力。

职责：

- 识别与分析
- 高亮与预览
- 建议与确认
- 业务面板数据组织

原则：

- 业务增强层不能侵入基础编辑主链路

---

## 8. 关键交互约束

### 8.1 标题栏

只放文档级动作：

- 文档标题
- 模式切换
- 扩展面板开关

不要放：

- 大量格式按钮
- 与当前文稿弱相关的系统级操作

### 8.2 快捷工具条

定位：

- 承接最高频的块级操作
- 是 slash menu 的显式捷径，不是完整替代品

建议保留：

- H1
- H2
- 引用
- 无序列表
- 有序列表
- 分割线

如存在业务增强模式，再追加场景专属入口，例如 `@` 实体引用。

### 8.3 纸面编辑区

要求：

- 始终是主视觉中心
- 保持单一连续阅读面
- 高频动作以浮动小按钮或贴边按钮存在

不建议：

- 多层卡片包裹正文
- 用管理后台式工具条包围正文
- 让多个面板同时争夺视觉中心

### 8.4 设置面

设置应当是独立 surface，而不是正文里的长弹窗。

建议收纳：

- 主题
- 排版模板
- 导入导出
- 工作区级偏好项

### 8.5 扩展助手面板

如果存量项目需要引入 AI 或结构化辅助，必须遵守以下边界：

- 助手负责展示和建议
- 推理能力按需触发
- 不自动改正文
- 不让基础交互依赖模型返回

---

## 9. 视觉语言建议

### 9.1 整体方向

视觉目标不是“炫”，而是“稳、轻、可久留”。

建议关键词：

- 克制
- 温和
- 留白
- 低刺激

### 9.2 设计 token 建议

至少抽出以下 token 层：

- 颜色：页面背景、表面背景、边框、正文、次级文本、强调色
- 间距：4 / 8 / 16 / 24 / 40
- 圆角：4 / 8 / 12 / 16
- 阴影：轻投影即可
- 字体：正文、标题、等宽字体

### 9.3 视觉落点

- 页面背景弱于纸面
- 纸面强于页面，但不做过重边框
- 侧栏和设置使用同一套表面语言
- 高亮、选中、激活尽量使用低饱和强调色

---

## 10. 扩展能力的设计边界

存量项目往往容易把“智能能力”直接堆到主界面里，这会破坏整套设计哲学。

正确做法是分两层：

### 10.1 本地稳定层

负责：

- 基础编辑
- 工作区切换
- 快速插入
- 高亮
- 轻量预览
- 本地规则识别

特点：

- 快
- 稳
- 可预测

### 10.2 推理建议层

负责：

- 解释
- 补全
- 冲突分析
- 摘要和推荐

特点：

- 按需调用
- 只给建议
- 不接管主链路

这是后续在存量项目中接 AI 时最重要的边界。

---

## 11. 存量项目迁移方案

### 阶段一：先建立壳层

目标：

- 完成工作区侧栏 + 文稿主区 + 设置 surface 的基础框架

输出物：

- 新编辑器页面骨架
- 统一 store
- 文稿切换与持久化

验收标准：

- 用户进入页面即可写
- 文件切换不丢内容
- 设置入口不影响正文主区

### 阶段二：接入纸面编辑体验

目标：

- 替换或封装现有编辑器，实现“正文优先”的可写区

输出物：

- Markdown 导入导出链路
- 轻量快捷工具条
- 标题栏与正文联动

验收标准：

- 用户不需要理解编辑态 / 预览态
- 高频块操作可在正文附近完成

### 阶段三：收纳低频能力

目标：

- 将主题、导入导出、模板偏好等收进设置面

输出物：

- 独立设置 surface
- 偏好项持久化

验收标准：

- 正文区明显更干净
- 低频功能仍然可达，但不常驻打扰

### 阶段四：叠加业务增强层

目标：

- 在不破坏正文主路径的前提下，加入业务专属辅助能力

输出物：

- 侧挂助手面板
- 轻量浮层预览
- 建议流与确认流

验收标准：

- 新能力不会挤压正文中心地位
- 用户可以忽略助手继续完成写作

---

## 12. 风险与规避

### 风险一：把迁移理解成换皮

问题：

- 只改配色和圆角，正文工作流仍然是旧逻辑

规避：

- 优先改结构和交互分层，再做视觉收口

### 风险二：把所有功能都放进标题栏和工具条

问题：

- 工具重新成为视觉中心

规避：

- 标题栏只放文档级动作
- 快捷工具条只放高频块操作

### 风险三：扩展模式做成第二套系统

问题：

- 模式切换后用户像进入另一个后台

规避：

- mode 只增加辅助能力，不改变正文主链路

### 风险四：AI 侵入基础编辑链路

问题：

- 编辑延迟、不稳定、不可预测

规避：

- 本地层负责实时反馈
- AI 只进入建议层

---

## 13. 最终验收标准

如果存量项目完成复刻，用户应该明显感受到以下结果：

- 打开页面后不用学习就能开始写
- 视线自然落在正文，而不是工具
- 常用动作离内容很近
- 低频功能在需要时能找到，但平时不干扰
- 即便增加高级能力，正文仍然是主角

如果只能用三个词概括最终目标，应当仍然是：

- 沉浸
- 克制
- 以内容为中心

---

## 14. 推荐落地顺序

建议按以下优先级实施，而不是并行大改：

1. 统一状态模型
2. 建立工作区 / 文稿 / 设置三层壳
3. 把正文做成真正的纸面主区
4. 收纳低频能力
5. 最后再叠加业务增强与智能建议

这个顺序的原因很简单：

**先稳定“写”，再补强“管”，最后才是“懂”。**
